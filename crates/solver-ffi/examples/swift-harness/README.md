# swift-harness — minimal Swift consumer of `solver-ffi`

A single-file Swift program that links against the `solver-ffi`
staticlib and calls every public FFI symbol. This is the dry run of
Poker Panel's consumption path — if this builds and runs, the contract
the Swift side will consume is real and loadable.

## What this proves

- `cbindgen` output (`../../include/solver.h`) is valid C that Swift's
  Clang importer can parse.
- Rust's `libsolver_ffi.a` exports every symbol declared in that
  header with the expected signature.
- The `repr(C)` struct layouts Swift infers from the header match the
  layouts Rust emits — i.e. `HandState` and `SolveResult` round-trip
  across the FFI boundary byte-for-byte.

## What this does **not** prove

- Correctness of the solver. The v0.1 stubs return non-OK status codes
  on purpose; this harness is about linkage and ABI, not logic.
- Apple framework integration. We use `swiftc` standalone, not Xcode.
  A successful build here is necessary but not sufficient for the
  eventual Poker Panel Xcode target.

## Build

From the repo root, first produce the static library:

```bash
cargo build --release -p solver-ffi
```

This generates:

- `target/release/libsolver_ffi.a`      (static — what Swift links)
- `target/release/libsolver_ffi.dylib`  (dynamic — for `dlopen` use)
- `crates/solver-ffi/include/solver.h`  (regenerated by `build.rs`)

Then, from this directory (`crates/solver-ffi/examples/swift-harness/`):

```bash
swiftc main.swift \
    -import-objc-header ../../include/solver.h \
    -L ../../../../target/release \
    -lsolver_ffi \
    -o swift-harness
```

Flag-by-flag:

| Flag | Why |
|---|---|
| `-import-objc-header ../../include/solver.h` | Makes every C decl in `solver.h` visible to Swift. For a proper Xcode target this would be a bridging header; for a standalone swiftc invocation this is the equivalent. |
| `-L ../../../../target/release` | Cargo's output dir for the staticlib. |
| `-lsolver_ffi` | Link against `libsolver_ffi.a` (swiftc resolves the `.a` automatically when the `.dylib` isn't preferred). |
| `-o swift-harness` | Name the resulting binary. |

### Sanity-check the header was freshly generated

If you suspect the header is stale:

```bash
rm crates/solver-ffi/include/solver.h
cargo build -p solver-ffi
ls -l crates/solver-ffi/include/solver.h   # should exist again
```

`build.rs` regenerates it from `src/lib.rs` on every build.

## Run

```bash
./swift-harness
```

Expected output with the current stubs:

```
poker-solver Swift harness — ABI smoke test
============================================
solver_version():          0.1.0-wip
solver_new():              <null>  (expected for v0.1 stub)
solver_solve():            status=-2          # InternalError — stub has no impl yet
  action_count            = 0
  hero_equity             = 0.0
  iterations              = 0
solver_lookup_cached():    status=1           # CacheMiss — stub always misses
============================================
ABI smoke test complete. All symbols linked and callable.
solver_free():             returned
```

Any status code from the documented `SolverStatus` enum is acceptable
here. What's not acceptable is a crash, a linker error, or any non-enum
integer value — those indicate the ABI is broken.

## Troubleshooting

**"ld: library not found for -lsolver_ffi"**
Run `cargo build --release -p solver-ffi` first; the staticlib has to
exist before `swiftc` can link it.

**"use of unresolved identifier 'HandState'"**
`solver.h` was not imported. Check the path to `-import-objc-header`.
It is relative to your current working directory.

**"Symbol not found: _solver_version"**
The `.a` exists but was built without the right symbols, usually because
`crate-type` in `Cargo.toml` doesn't include `staticlib`. Check the
workspace: it should list `crate-type = ["staticlib", "cdylib", "rlib"]`.

**"error: unknown type name 'HandState'" during C parse**
The header regeneration failed at build time (cbindgen warning in the
cargo output). Run `cargo clean -p solver-ffi && cargo build -p solver-ffi`
and look for `cargo:warning=` lines explaining why.
